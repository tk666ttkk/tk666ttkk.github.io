List类型：

和java的LinkedList类似，常用来存储一个有序数据，可以看做一个双向链表结构，可正向和反向检索。

Redis的List类型通常用于存储具有时间顺序的数据，例如日志、消息队列等。

特征：

  1. 有序：Redis List是有序的数据结构，它按照元素插入的顺序来存储数据。你可以在列表的两端执行插入和删除操作。
  2. 元素可以重复：与集合（Set）不同，Redis List允许元素重复出现。这意味着你可以在同一个List中多次存储相同的元素。
  3. 插入和删除快：Redis List对于在列表的两端执行插入和删除操作非常快速。这是因为它是一个双向链表结构，使得在列表的头部或尾部添加或删除元素的时间复杂度是O(1)。
  4. 查询速度一般：虽然Redis List对于插入和删除操作非常高效，但对于随机访问（根据索引或值查找元素）而言，速度较慢。这是因为在链表中进行随机访问需要遍历链表，时间复杂度是O(n)。

Set类型：

和java的HashSet类似，value为null的HashMap，因此也是一个Hash表

Redis的Set类型通常用于存储不重复的元素集合，例如用户的标签、喜好、关注列表等。

特征：

  1. 无序：Redis Set是无序的，这意味着元素在集合中的存储顺序不是固定的，与元素的插入顺序无关。
  2. 元素不可重复：与List不同，Set类型不允许重复元素，每个元素在集合中只能出现一次。
  3. 查找快：Redis Set内部是通过哈希表实现的，因此查找元素的速度非常快，时间复杂度是O(1)。
  4. 支持并集、差集和交集功能：Redis提供了一系列用于集合操作的命令，如并集（SUNION）、差集（SDIFF）和交集（SINTER）等，这些命令可以用于对多个集合进行操作，如合并、比较和筛选等

SortedSet类型：

可排序的Set集合，和java的TreeSet类似，但TS底层是红黑树，而SortedSet中的每一个元素都带有一个score属性，可以基于score属性对元素排序，底层的实现是一个跳表（SkipList）加Hash表

特征:

  1. 可排序：Redis Sorted Set是一个可排序的集合，它根据每个元素的分数（score）属性对元素进行排序。这允许你按照特定顺序访问元素，无论是升序还是降序。
  2. 元素不重复：与Set类型类似，Sorted Set中的元素是不重复的，每个元素只能出现一次。
  3. 查询速度快：Redis Sorted Set的底层实现是一个跳表（Skip List）和一个哈希表（Hash Table）的结合，这使得查找元素的速度非常快，时间复杂度是O(log(N))。
  4. 每个元素都有一个分数（score）属性：与普通Set不同，Sorted Set中的每个元素都关联有一个分数，这个分数可以是任意浮点数值，用于排序和比较元素。

Redis的Sorted Set常用于需要按分数排序的场景，如排行榜、计分系统、优先级队列等。

Lettuce:

Lettuce是一个Java编写的高性能、异步的Redis客户端库，用于与Redis服务器进行交互。它被设计成异步和非阻塞的，允许开发者在高并发应用中高效地与Redis数据库进行通信。

  1. 异步和非阻塞：Lettuce是一个异步、非阻塞的Redis客户端，它允许你执行Redis操作而不会阻塞应用程序的执行线程。这对于高并发的应用程序非常有用，因为它可以减少线程等待时间，提高性能。

  2. 连接池：Lettuce内置了连接池支持，这使得在多个线程之间共享Redis连接变得容易，同时能够有效地管理和重用连接。

  3. 集成Redis Sentinel和Cluster：Lettuce可以轻松地与Redis Sentinel和Redis Cluster一起工作，支持高可用性和分布式部署。

  4. 支持SSL/TLS：Lettuce支持通过SSL/TLS加密与Redis服务器之间的通信，提供数据安全性。

  5. Reactive编程模型：Lettuce还提供了对响应式编程模型的支持，可以与Spring Framework的Project Reactor等库集成，允许开发者编写异步和响应式的Redis操作。

Reddison实现可重入锁：

可重入锁（Reentrant Lock）是一种支持同一个线程多次获取同一把锁的锁机制。其原理涉及到两个关键概念：锁的持有数（hold count）和线程关联。

获取一次锁，Hash结构存储获取到锁的线程和获取的次数。field 改为当前线程，value+1，释放一次锁，value-1，当value=0，即表示所有的业务已经完成，最外层的业务可以释放锁。

以下是可重入锁的原理：

  1. 锁的持有数（hold count）：可重入锁跟踪了每个线程对锁的获取次数。当一个线程第一次获取锁时，锁的持有数从0增加到1，每次重复获取锁时，持有数都会递增。只有当锁的持有数归零时，锁才会被完全释放。
  2. 线程关联：可重入锁需要知道哪个线程当前持有了锁。为了实现这个功能，每个线程都会与锁关联起来，通常是通过线程本地存储（Thread-Local Storage）来实现。这意味着每个线程都有自己独立的锁持有数，线程之间互不干扰。
  3. 获取锁：当一个线程尝试获取锁时，它会首先检查锁是否已经被其他线程持有。如果锁没有被其他线程持有，那么它会被分配给当前线程，并将锁的持有数设置为1。如果锁已经被当前线程持有，那么只会增加持有数，而不会重新分配锁。这就是可重入的关键点，同一个线程可以多次获取同一把锁，只要它在每次获取后都会释放。
  4. 释放锁：当一个线程释放锁时，它会将锁的持有数减一。只有当持有数归零时，锁才会完全释放，可以被其他线程获取。
  5. 保护共享资源：可重入锁的主要目的是保护共享资源，确保多个线程可以安全地访问这些资源。当一个线程持有锁时，其他线程如果尝试获取锁，将会被阻塞，直到锁被释放。
  6. 重入性：如果一个线程已经获得了锁，并且多次请求获取锁，计数器会累加，但只有当线程完全释放锁时，其他线程才有机会获取锁。
  7. 线程安全：可重入锁需要确保计数器的操作是线程安全的，通常使用原子操作来实现。计数器（Counter）：可重入锁维护一个计数器，用于记录当前线程获取锁的次数。初始时，计数器为0。

主从数据同步的原理：增量同步和全量同步

全量同步：第一次连接master和slave宕机太久导致数据都没来得及读取

第一阶段:携带replid和offset向master请求增量同步，判断是否是第一次，即判断replid是否相同，否则返回master的replid和offset给slave保存版本信，下次来只要对比replid即可知道该节点是否是第一次访问。第一次拒绝增量同步，后续进行增量同步；

如何判断是否是第一次呢？

Replication Id：replid，数据集的标记，slave会继承master的id

offset：偏移量，随着repl_baklog文件的数据增多而逐渐增大，slave完成同步的时候也会记录当前同步的offset，若slave的offset小于master的，则说明版本落后，需要更新。

​

第二阶段：生成RDB文件，执行bgsave，发送给slave，slave清空本地数据，加载新的RDB文件。这里第一次来肯定是进行全量同步。

第三阶段：在RDB期间开启独立的线程，记录RDB期间收到的命令，repl_baklog文件。发送给slave，slave接受并执行，保持和master之间的同步。

增量同步：slave节点断开创造性回复并且能找到slave的offset，即slave的offset没有因master的读取而被覆盖。当slave重启后同步，则进行增量同步。

判断replid，不是第一次，continue。

进行增量同步，repl_baklog大小有上限，写满会覆盖最早的数据。master和slave在其中会有各自的offset记录最新同步的位置。如果slave断开的时间过久，导致未北方的数据被覆盖，则无法基于log做增量同步，只能再次进行全量同步。

数据倾斜：无法使数据分片的内存资源达到均衡

redis阻塞:对元素较多的集合做运算，会耗时较久，主线程阻塞

CPU压力:序列化反序列化，高使用率CPU。

如何发现BIgkey：

1.redis-cli –bigkeys

扫描出每一种数据类型占用内存最多的key，不全。

2.scan扫描

自己写程序扫描：scan offset：offset:开始是0，每次会返回偏移量和key，background运行扫描key，扫描出超过指定大小的不同的数据类型的所有key。

redis3：扫描遍历，逐个删除子元素，最后删除bigkey。SSCAN，HSCAN，ZSCAN，SCAN。

redis4：unlink

hash-max-ziplist-entries:配置entry上限，entry过多导致bigkey。
